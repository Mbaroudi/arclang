## 1. Repository Overview

The elk-models repository contains test models for the Eclipse Layout Kernel (ELK) project, organized into four major categories:

### Repository Structure

| Folder | Description | File Count | Purpose |
|--------|-------------|------------|---------|
| `examples/` | Example graph specifications illustrating common ELK configurations | 45 .elkt files | Educational examples of key features |
| `tests/` | Test graphs written/generated by ELK developers | 193 .elkt files | Comprehensive test coverage for algorithms |
| `tickets/` | Graphs reproducing GitHub issue reports | 110 .elkt files | Bug reproduction and regression tests |
| `realworld/` | Real-world graphs from Ptolemy and other systems | ~50+ files | Complex real-world use cases |

**Total Examples:** 348+ .elkt files demonstrating ELK features

### File Formats
- `.elkt` - Textual ELK graph language (primary format)
- `.elkg` - EMF's XMI-based representation
- `.json` - JSON format for ELK graphs

## 5. Example Code Analysis

### 5.1 `examples/general/mixingDirection.elkt`

**Demonstrates:** Different layout directions at different hierarchy levels

```elk
node outsideTopToBottom {
    elk.direction: DOWN
    nodeLabels.placement: "H_LEFT V_TOP OUTSIDE"
    
    node leftToRight {
        elk.direction: RIGHT
        node n1
        node n2
        edge n1->n2
    }
    node bottomToTop {
        elk.direction: UP
        node n1
        node n2
        edge n1->n2
    }
    
    edge bottomToTop -> leftToRight
}
```

**Key Insights:**
- Each hierarchical level can have independent `elk.direction`
- Options: RIGHT, LEFT, UP, DOWN
- Label placement adapts to direction
- Edges between different-direction containers work correctly

**ArcLang Application:**
- Architecture diagrams often mix flow directions
- Top-level: horizontal system flow
- Component internals: vertical or custom
- Critical for Capella's mixed layout needs

---

### 5.2 `examples/hierarchy/hierarchicalLayoutMixing.elkt`

**Demonstrates:** Different layout algorithms at different hierarchy levels

```elk
algorithm: elk.layered
hierarchyHandling: INCLUDE_CHILDREN

node outside {
    node inside {
        // Uses parent's elk.layered
    }
    
    node insideOtherLayout {
        algorithm: elk.stress
        desiredEdgeLength: 50
        elk.padding: "[top=10,left=10,bottom=10,right=10]"
    }
    
    node insideOtherDirection {
        elk.direction: DOWN
        hierarchyHandling: SEPARATE_CHILDREN  // Required for direction change
    }
}
```

**Key Insights:**
- `hierarchyHandling: INCLUDE_CHILDREN` - layered processes whole graph at once
- Can override algorithm per node: `elk.layered`, `elk.stress`, `elk.force`, `elk.box`, `elk.rectpacking`
- Changing direction requires `hierarchyHandling: SEPARATE_CHILDREN`
- Separates hierarchical edges from internal layouts

**ArcLang Application:**
- Root: layered for system architecture
- Components: box layout for internal organization
- Subsystems: force-directed for loose coupling
- Provides flexibility for different diagram types

### 5.4 `examples/general/spacing/portsSurrounding.elkt`

**Demonstrates:** Port surrounding margins and alignment strategies

```elk
spacing.portPort: 5
spacing.portsSurrounding: "[top=20.0,left=5.0,bottom=0.0,right=15.0]"

node inner {
    layout [ size: 100, 100 ]
    portConstraints: FIXED_SIDE
    
    portAlignment.west: BEGIN     // Starts at top margin
    portAlignment.east: CENTER    // Centered in allowed span
    portAlignment.north: JUSTIFIED  // Evenly distributed
    portAlignment.south: DISTRIBUTED  // Distributed with outer padding
    
    port pw1 { ^port.side: WEST }
    port pw2 { ^port.side: WEST }
    // ... more ports
}

// Override per node
node inner2 {
    spacing.individual: "spacing.portsSurrounding:[top=40.0,left=10.0,bottom=10.0,right=30.0]"
    // ... same structure
}
```

**Key Insights:**
- `spacing.portsSurrounding` defines allowed port placement area
- Semantics: `top` = upper part of E/W borders, `left` = left part of N/S borders
- Works with `portAlignment`: BEGIN, CENTER, JUSTIFIED, DISTRIBUTED
- Can override per-node with `spacing.individual`
- `spacing.portPort` defines minimum spacing between adjacent ports

**ArcLang Application:**
- Control where on a side ports can be placed
- Avoid ports too close to corners
- Create symmetric port layouts
- Essential for clean interface visualization

### 5.5 `examples/ports/portConstraints.elkt`

**Demonstrates:** Different port constraint strategies

```elk
// Free positioning - algorithm decides everything
node freePorts {
    node n1 {
        portConstraints: FREE
        port p1
        port p2
    }
}

// Fixed side - ports must stay on specified sides
node fixedSide {
    node n1 {
        portConstraints: FIXED_SIDE
        port p1 { ^port.side: WEST }
        port p2 { ^port.side: EAST }
    }
}

// Fixed position - exact coordinates
node fixedPosition {
    node n1 {
        portConstraints: FIXED_POS
        port p1 { layout [ position: -5, 5 ] }   // Can extend beyond node
        port p2 { layout [ position: 35, 10 ] }
    }
}
```

**Key Insights:**
- `FREE` - algorithm optimizes port positions for edge routing
- `FIXED_SIDE` - port stays on specified side (NORTH/SOUTH/EAST/WEST)
- `FIXED_POS` - exact pixel coordinates (relative to node corner)
- `FIXED_ORDER` (not shown) - maintains port order on side
- Fixed position can extend beyond node boundaries (negative values)

**ArcLang Application:**
- Capella interfaces are FIXED_SIDE (logical component boundaries)
- User-placed ports need FIXED_POS
- Order matters for grouped interfaces (FIXED_ORDER)
- Critical for maintaining architectural meaning

## 8. Key ELK Concepts for ArcLang

### 8.1 Hierarchy Handling Modes

**Critical Decision Point:** How to process hierarchical graphs

| Mode | Behavior | Use Case |
|------|----------|----------|
| `SEPARATE_CHILDREN` (default) | Process each level bottom-up independently | Different directions per level |
| `INCLUDE_CHILDREN` | Process entire graph at once | Hierarchical edges, unified layout |

**Implication:** Capella needs INCLUDE_CHILDREN for proper component interface routing.

### 8.2 Port Constraint Hierarchy

**Increasing constraint levels:**

1. `FREE` - Algorithm decides everything
2. `FIXED_SIDE` - Side is fixed, position on side is free
3. `FIXED_ORDER` - Side and order fixed, exact position free
4. `FIXED_POS` - Everything fixed

**Implication:** Capella typically uses FIXED_SIDE (logical side) with potential FIXED_ORDER for grouped interfaces.

### 8.3 Spacing System Architecture

ELK has a comprehensive spacing system:

- **Between elements:** `spacing.nodeNode`, `spacing.edgeNode`, `spacing.edgeEdge`
- **Layer-specific:** `spacing.*BetweenLayers` (horizontal in LEFT/RIGHT)
- **Port-specific:** `spacing.portPort`, `spacing.portsSurrounding`
- **Container-specific:** `elk.padding`
- **Label-specific:** `spacing.labelNode`, `spacing.labelPort`
- **Override mechanism:** `spacing.individual`

**Implication:** Need systematic approach to expose these without overwhelming users.

### 8.4 Algorithm Selection Strategy

| Algorithm | Best For | Characteristics |
|-----------|----------|-----------------|
| `elk.layered` | Directed flow graphs | Layered, directional, hierarchical |
| `elk.box` | Simple containers | Packing, minimal computation |
| `elk.force` | Undirected graphs | Force-directed, organic |
| `elk.stress` | Circular layouts | Stress minimization |
| `elk.rectpacking` | Disconnected components | Rectangle packing |

**Implication:** ArcLang should default to `elk.layered` but allow mixing for different diagram areas.

### 8.5 Label Placement System

**Three-part specification:** `"H_ALIGNMENT V_ALIGNMENT INSIDE|OUTSIDE [H_PRIORITY]"`

Horizontal: `H_LEFT`, `H_CENTER`, `H_RIGHT`  
Vertical: `V_TOP`, `V_CENTER`, `V_BOTTOM`  
Location: `INSIDE`, `OUTSIDE`  
Priority: `H_PRIORITY` (prefer horizontal placement)

**Implication:** Need clear UI for common combinations, advanced mode for full control.

## 11. Potential Pitfalls for ArcLang

### 11.1 Known Issues from Tickets

| Issue Pattern | Description | Mitigation |
|---------------|-------------|------------|
| Hierarchical port overlap | Ports placed on top of each other | Test with `tickets/layered/193_*` |
| Self-loop crashes | Various NPE and AIOOBE with self-loops | Use latest ELK version, test extensively |
| Port label positioning | Labels overlap nodes/edges | Configure `portLabels.placement` carefully |
| Interactive + hierarchy | Interactive mode breaks with hierarchies | Test mixed mode early |
| Spline routing bugs | Splines overlap nodes | Use ORTHOGONAL as default |

### 11.2 Performance Considerations

**From real-world examples:**

- Ptolemy examples have 50-200 nodes
- Deep nesting (4-5 levels) increases complexity
- INCLUDE_CHILDREN mode is slower but more accurate
- Interactive mode needs multiple layout passes

**Recommendation:** 
- Profile early with large examples
- Provide progress feedback for slow layouts
- Consider incremental layout for interactive editing

### 11.3 API Complexity

**ELK has 100+ layout options:**

- Need careful curation for ArcLang users
- Presets for common use cases
- Advanced mode for power users
- Good defaults critical
